# Explanation: Declarative Execution Security

One of the most persistent risks when delegating filesystem and execution privileges to Large Language Models (LLMs) is unintended system mutation, primarily through arbitrary **shell injection**.

Traditionally, systems hook Agents into OS environments by exposing a raw `bash` or `sh` execution tool. However, if an agent hallucinates, or is maliciously prompted with a destructive command (e.g., `feature-branch; rm -rf /`), passing this string directly into `bash -c` allows the intermediary shell to interpret semi-colons, variable substitutions, and operators catastrophically.

`dux` solves this specifically via **Declarative Binary Tools**.

## How Shell-Bypass Protects Execution

When defining a tool utilizing the `binary:` configuration schema, you dictate a strict execution pipeline unmediated by an OS shell:

1.  **JSON Schema Validation**: Standard inputs explicitly bound to precise types (strings, ints).
2.  **String Slice Allocation**: Arguments are parsed as discreet arrays in Go (e.g. `[]string{"push", "origin", "my-branch"}`).
3.  **Kernel-level Invocation**: The array is dispatched directly to the POSIX `execve` system call mapping (`os/exec` in Go). 

Because `execve` simply passes arguments to the specified process memory space without structural parsing:
* `git commit -m "update; chmod 777 /"` results in a git commit literally named `"update; chmod 777 /"`. 
* The injection attempt is neutralized safely as inert string data.

## Why use Substitution instead of Template Languages?

You will note `dux` enforces extremely simple `{key}` substring replacements over features like `text/template` or `Jinja`.

While template languages offer conditionals and loops, they reintroduce ambiguity parsing inside the application mapping phase. A rogue JSON attribute supplied by the LLM (`{{ if panic }}...`) could arbitrarily crash execution if the template renderer is exploited.

Basic substring replacement guarantees safe and linear parameter assignment, isolating complexity entirely inside the `binary` constraints.

## Coupling with Google CEL

Declarative Tool definitions naturally complement hitl execution (`Human-in-the-Loop`).

By attaching `requirements.supervision` to a Tool using a [Common Expression Language (CEL)](https://github.com/google/cel-spec) string:

```yaml
requirements:
  supervision: "args.target == 'production'"
```

`dux` natively validates the inputs generated by the LLM against your defined safe-operational policy prior to the POSIX `execve` call. This gives users scalable, contextically-aware autonomy that fails predictably to human-review only on unsafe bounds.
